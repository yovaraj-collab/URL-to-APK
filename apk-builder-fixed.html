🔨 Option 1 — Build with Android Studio
01
Download and Extract the ZIP

Click Download ZIP above and extract it to a new folder. Keep the folder structure exactly as exported — the top-level folder is your Android project root.

02
Install Android Studio

Install Android Studio with the Android SDK, platform tools, and an emulator. This is the easiest way to open, sync, build, sign, and debug the project.

03
Open the Project and Let Gradle Sync

In Android Studio, choose File → Open and select the extracted folder. Wait for Gradle sync. If prompted to install a missing SDK or build tools version, allow it.

04
Apply Your App Icon

Right-click app/src/main/res → New → Image Asset. Android Studio will generate launcher icon files for all required densities automatically.

05
Export an APK

Use Build → Build Bundle(s) / APK(s) → Build APK(s) for a quick test build, or Build → Generate Signed Bundle / APK for a signed release APK. For Play Store use an .aab instead.

06
Test on a Device or Emulator

Confirm the site loads correctly, navigation works, permissions behave as expected, and icon, splash color, and orientation match your intended release.

🤨 Option 2 — Build with GitHub + GitHub Actions
07
Download & Extract the ZIP First

Save and extract the ZIP before uploading to GitHub. GitHub Actions needs the actual project files — app/, gradle/, build.gradle, settings.gradle, gradlew — present in the repository.

08
Create a GitHub Repository and Upload

Create a new repository, then upload the extracted project contents into the root. The top level should directly show gradlew, settings.gradle, and the app folder.

09
GitHub Actions Workflow (included)

The ZIP already contains .github/workflows/android-build.yml. It checks out the repo, installs Java 17, caches Gradle, and runs ./gradlew assembleDebug on every push. The resulting APK is uploaded as a workflow artifact.

10
Signed Release Builds

For release APK/AAB, store your keystore as a Base64 secret in GitHub Secrets along with the keystore password, alias, and key password. Update the workflow to decode the keystore and pass signing config to Gradle.

11
Create the .jks Keystore Once

Generate the keystore locally in Android Studio via Build → Generate Signed Bundle / APK → Create new, or with the Java keytool command. Reuse the same keystore for all future updates — Google Play requires consistent signing.

12
Host a Privacy Policy via GitHub Pages

Google Play requires a public privacy policy URL. Enable GitHub Pages in repository settings and publish a simple static page. Use that URL in your Play Console listing.

🚀 Final Step — Publish to Google Play Store
13
Create a Google Play Developer Account

Register at Google Play Console and complete all account setup steps. Keep your developer profile consistent with your business, brand, and privacy policy details.

14
Prepare a Signed AAB

Option 1: generate the signed .aab in Android Studio. Option 2: let GitHub Actions build it using your stored keystore. Keep the .jks, alias, and passwords safely backed up forever.

15
Create the App Entry in Play Console

Select Create app, enter your app name and language, declare whether it is an app or game, and complete the initial declarations.

16
Complete Store Listing & App Content

Add descriptions, screenshots, icon, feature graphic, contact details, privacy policy URL, target audience, ads declaration, and data safety responses. Incomplete sections are the most common cause of first-submission delays.

17
Upload, Review & Roll Out

Upload the signed .aab in Production → Create new release. Submit for review, monitor policy warnings, and roll out carefully. For every future update: increment version code, rebuild a signed AAB with the same keystore, publish a new release.

⚡ How to choose: Use Option 1 for the simplest path. Use Option 2 for automated cloud builds via GitHub Actions. The generated ZIP already includes the GitHub Actions workflow file.
🤱 Good practice: APK for local testing and sideloading. AAB for Play Store. Keep the same package name, signing key, and version identity across all future versions.
⚠️ Important: A WebView wrapper alone may face Play policy or quality review issues. Your app should provide real utility, use a domain you own, include a valid privacy policy, and clearly explain what value the mobile app adds beyond simply opening a website.

Google Play Console — Payment & Future Updates

📌 Google Play Developer Account — Important Notes ►

Google Play Developer registration is a one-time payment. You do not pay again for future app updates under the same developer account.

First-Time Payment to Publish on Google Play

Google charges a one-time registration fee when creating the Play Console developer account. After approval, you can publish apps without paying again.

  1. Create or sign in with the Google account that will own the app.
  2. Go to Google Play Console and register as a developer.
  3. Pay the one-time developer registration fee.
  4. Complete account verification steps requested by Google.
  5. Create your app listing inside Play Console.
  6. Upload your signed .aab file for release. APK can be used for testing, but Play Store production normally uses AAB.

How Updates Are Sent to Existing Devices

Future updates do not require a new Play Console account or another registration payment.

  1. Make changes in your project and generate a new build.
  2. Increment versionCode (and usually versionName).
  3. Build a new signed APK/AAB using Option 1 or Option 2.
  4. Upload the new signed .aab to the same existing app in Google Play Console.
  5. Submit the release to the chosen track (Internal, Closed, Open, or Production).
  6. After Google approves, devices receive the update automatically or via the update button in Play Store.

Important Rules for Future Updates

  1. Same package name — required for the new release to update the existing app on devices.
  2. Consistent signing — signing setup must stay aligned with the original release.
  3. Higher versionCode every release — Google Play rejects uploads with the same or lower versionCode.
  4. Prefer AAB for Play Store — Android App Bundle is the standard format for Play Store distribution.

The .jks Keystore and Updates

Create the .jks keystore once locally before the first release. For Option 2, convert it to Base64 and store it in GitHub Secrets. During each GitHub Actions run, reconstruct the keystore temporarily for signing. Reuse the same keystore for all future releases. If the signing identity changes incorrectly, the new build will not be accepted as an update for the existing Play Store app.

GitHub's Role vs. Google Play's Role

GitHub stores code and automates builds. Google Play Console is the official release and update channel for end users. GitHub Actions cannot publish to Play Store automatically unless you add a separate deployment workflow using a Google Play service account.
